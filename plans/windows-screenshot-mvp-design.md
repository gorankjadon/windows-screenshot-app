## Problem Statement
Build a Windows-first desktop screenshot app that captures the full screen, a selected region, or the active window, then saves a PNG locally and copies the image to the clipboard. The app should feel lightweight, fast, and local-first, with no cloud sync, no account system, and no AI features in v1.

## What Makes This Cool
The whole point is speed and absence of ceremony. One shortcut should turn what is on screen into a PNG on disk and an image in the clipboard without opening an editor, prompting for an account, or asking the user to manage a library.

## Constraints
- Windows-first desktop app
- MVP only
- Must capture full screen, selected region, and active window
- Must save PNG files locally
- Must copy the capture to the clipboard
- Must let the user choose the save folder
- Must support configurable keyboard shortcuts
- Must be lightweight and fast
- Must not include cloud sync, accounts, or AI in v1

## Premises
1. The user value is "capture instantly and get out of the way," not editing, sharing, or organizing.
2. A native Windows stack is a better fit than a cross-platform stack for hotkeys, capture APIs, and startup performance.
3. Simultaneous "save to disk + copy to clipboard" is a core behavior, not an optional convenience.
4. A tiny background app with a minimal settings window is enough for v1; a gallery, editor, or onboarding flow is not required.
5. The narrowest usable MVP is a utility for people who already know what screenshots are and just want a faster default.

## Approaches Considered
### Approach A: Native Utility MVP
Summary: Build a small native Windows app that lives in the tray, registers global hotkeys, captures the requested area, saves a PNG to a chosen folder, and copies the image to the clipboard.

Effort: M
Risk: Low
Pros:
- Smallest surface area that still satisfies all requirements
- Best shot at low memory use and fast startup
- Uses mature Windows APIs for hotkeys, clipboard, and screen capture
Cons:
- Windows-only by design
- Basic UI only; not visually modern
- Active-window capture may need careful handling for edge cases like shadows and DPI
Reuses:
- .NET desktop runtime
- Win32 hotkey registration
- Standard Windows clipboard and folder selection APIs

### Approach B: Modern Windows App SDK Version
Summary: Build the same product idea with WinUI 3 and newer Windows App SDK primitives for a more modern shell and future polish.

Effort: M/L
Risk: Medium
Pros:
- More modern Windows UI foundation
- Better long-term runway if the app grows beyond MVP
- Easier to evolve into a polished packaged app
Cons:
- Higher implementation friction for a utility app MVP
- More moving parts before the first useful build
- Spends complexity on shell polish instead of capture speed
Reuses:
- .NET 8
- Windows App SDK packaging and settings patterns

### Approach C: Cross-Platform Shell with Native Capture Bridges
Summary: Use a cross-platform desktop shell such as Tauri or Electron, then bridge into native Windows capture and hotkey behavior.

Effort: L
Risk: High
Pros:
- Future cross-platform option if product direction changes
- Web UI development can be fast for settings screens
- Large ecosystem
Cons:
- More overhead for a Windows-only utility
- Hotkeys and low-latency capture become more fragile
- Solves a future problem the MVP does not have
Reuses:
- Web tooling ecosystems
- Existing desktop shell packaging

## Recommended Approach
Choose Approach A.

This is the narrowest MVP that still feels like a real product instead of a prototype. The product promise is simple: press a hotkey, get a screenshot saved locally, and have it ready to paste immediately. A native Windows utility is the most direct way to honor that promise.

CEO review mode: SCOPE REDUCTION.

CEO review conclusion: the product should be an instant local capture utility, not a screenshot platform.

## Proposed Tech Stack for Windows
- Language/runtime: C# on .NET 8
- Desktop UI: WPF
- Background presence: tray icon via Windows Forms `NotifyIcon` interop
- Global shortcuts: Win32 `RegisterHotKey`
- Screen capture: Win32 and GDI-based capture for v1, with foreground-window bounds from Win32/DWM APIs
- Clipboard: WPF clipboard APIs
- Image encoding: PNG via .NET imaging APIs
- Settings storage: JSON file in `%AppData%\<AppName>\settings.json`

Why this stack:
- It is the most boring, native, low-risk path for a Windows utility.
- It fits the real problems here: global hotkeys, screen bounds, clipboard, and local files.
- It avoids spending scope on cross-platform concerns the MVP does not need.

Implementation note from engineering review:
- For v1, define active-window capture as the current foreground window bounds as visible on screen.
- If dogfooding shows that occlusion-free capture is a hard requirement, revisit a more advanced capture path in v2.

## Narrowest MVP
The narrowest MVP is a background screenshot utility with one small settings window and three independent capture actions:

1. Full screen capture
2. Selected region capture
3. Active window capture

For every capture:
- Save a PNG in the configured folder
- Copy the image to the clipboard
- Use the configured hotkey

The settings window only needs:
- Save folder picker
- Three hotkey inputs
- Optional app name/version display
- Default shortcut: `Ctrl+Alt+S` for region capture

Recommended defaults locked by review:
- No launch-at-login in v1
- No success notification in v1
- Cursor excluded from captures
- If a capture fails, show a small error notification and save nothing

Explicitly out of scope for v1:
- Annotation or editing
- Capture history or gallery
- Upload/share links
- OCR or AI
- Cloud sync
- Accounts
- Multi-step workflows/macros

## Suggested Product Defaults
- Filename format: `Screenshot_YYYY-MM-DD_HH-mm-ss.png`
- Default activation: `Ctrl+Alt+S` opens region capture immediately
- Region capture flow: dimmed full-screen overlay with click-drag-select and Escape to cancel
- After capture: silent success by default, with a lightweight toast only on failure
- Multi-monitor behavior:
  - Full screen captures the full virtual desktop
  - Region capture can span monitors
  - Active window captures the current foreground window bounds

## System Architecture
```text
+-------------------+         WM_HOTKEY         +----------------------+
| Global Hotkeys    | -----------------------> | Capture Orchestrator  |
| Register/Unreg    |                          | route by capture type |
+-------------------+                          +----------+-----------+
                                                         |
                              +--------------------------+--------------------------+
                              |                          |                          |
                              v                          v                          v
                   +-------------------+      +-------------------+      +-------------------+
                   | Full Screen       |      | Region Overlay    |      | Active Window     |
                   | Capture Service   |      | + Crop Service    |      | Capture Service   |
                   +---------+---------+      +---------+---------+      +---------+---------+
                             \__________________________|__________________________/
                                                        |
                                                        v
                                              +-------------------+
                                              | Image Pipeline    |
                                              | bitmap -> PNG     |
                                              +----+----------+---+
                                                   |          |
                                                   v          v
                                     +------------------+   +------------------+
                                     | File Save        |   | Clipboard Copy   |
                                     | chosen folder    |   | image available  |
                                     +------------------+   +------------------+
```

## User Flow
```text
[App running in tray]
      |
      +--> [User presses hotkey]
                |
                +--> Full screen  ------+
                |                       |
                +--> Active window -----+--> [Capture image]
                |                       |
                +--> Region ------------+--> [If region: drag/select or Esc cancel]
                                               |
                                               +--> [Encode PNG]
                                                       |
                                                       +--> [Save file]
                                                       |
                                                       +--> [Copy to clipboard]
                                                       |
                                                       +--> [Silent success]

Failure paths:
- hotkey registration fails -> user sees conflict error in settings
- save fails -> user sees folder/permission error
- clipboard fails -> user sees clipboard busy error
- region cancelled -> no file, no clipboard change, no error
```

## Open Questions
- For active-window capture, is "foreground window as currently visible on screen" acceptable for v1, or do we need stricter occlusion-free capture?
- Should failures show Windows toast notifications or tray balloon notifications?

## Success Criteria
- The app starts quickly and sits quietly in the background
- Global hotkeys work while the settings window is unfocused
- Full screen, region, and active-window captures all produce valid PNG files
- Every successful capture is also present on the clipboard
- The save folder persists across restarts
- Shortcut changes persist across restarts
- The app feels faster and simpler than opening Snipping Tool and manually saving

## Engineering Plan
### Modules
- `AppShell`: startup, tray icon, settings window lifetime
- `SettingsStore`: load and save JSON settings with defaults
- `HotkeyManager`: register, unregister, validate conflicts
- `CaptureOrchestrator`: dispatch capture request and run save and clipboard pipeline
- `CaptureServices`: full screen, region, active window
- `NotificationService`: error-only user feedback

### Boundaries
- Keep Win32 interop thin and isolated in adapter classes
- Keep bitmap, file save, and clipboard pipeline in one place to avoid duplicate image handling
- Avoid a plugin architecture or generic command bus in v1

## Error And Rescue Map
| Failure | Cause | User Sees | System Behavior | Test? |
|---|---|---|---|---|
| Hotkey conflict | Shortcut already registered by another app | Validation error in settings | Old shortcut remains active | Yes |
| Save folder invalid | Missing path or deleted folder | Folder error notification | Capture discarded | Yes |
| Save permission denied | Folder not writable | Permission error notification | No partial success in v1 | Yes |
| Clipboard busy | Another process locks clipboard | Clipboard error notification | File save still succeeds | Yes |
| Region cancelled | User presses Escape or zero-area drag | Nothing | No file, no clipboard mutation | Yes |
| No foreground window | OS returns no usable window | Capture failed notification | No file, no clipboard mutation | Yes |
| DPI mismatch | Multi-monitor scaling math wrong | Wrong crop or window bounds | Treat as bug and test bounds math | Yes |

## Test Plan
### Unit tests
- Settings load and save with defaults and invalid JSON fallback
- Hotkey parsing, duplicate detection, reserved-key rejection, unregister and re-register behavior
- Filename generation and timestamp formatting
- Virtual-screen and crop-rectangle math across multiple monitors and DPI-scaled bounds
- Capture orchestrator behavior for success, region cancel, save failure, and clipboard failure

### Manual smoke tests
- Full-screen capture on single-monitor and multi-monitor setups
- Region capture across monitor boundaries
- Active-window capture on Explorer, browser, and a DPI-scaled app
- Folder change persists after restart
- Hotkey change persists after restart

## Performance Notes
- Capture once per action and reuse the same in-memory image for file save and clipboard copy
- Dispose unmanaged bitmap handles immediately after save and copy
- Keep the app single-process and background-idle when not capturing

## NOT in Scope
- Launch at login: useful, but not required to prove the capture loop
- Capture history or gallery: adds browsing complexity without helping the core loop
- Annotation or editing: turns a utility into a mini design app
- Upload or share links: adds network scope and account pressure
- OCR or AI: unrelated to the core capture promise
- Cross-platform support: spends effort solving a problem the product does not yet have
